# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A bot that plays chess on Lichess with human-like behaviour: neural-network move selection blended with Stockfish analysis, screen-capture board vision (OpenCV + template matching), and mouse automation. Entry point is `main.py`, which delegates to `clients/mp_original.py`.

## Python environment

Use the repo virtualenv: `venv/bin/python` (Python 3.12). There is no bare `python` on PATH, and the system `python3` lacks the dependencies.

## Lint

`venv/bin/ruff check <changed files>` (config in `ruff.toml`; stylistic rules that clash with existing code idioms are disabled). The repo has ~170 pre-existing violations (mostly unused imports/variables), so lint only the files you touched, not the whole tree.

## Verifying changes

The bot cannot be run end-to-end here — it needs an X11 display and a live Lichess page. Never assume a live run is possible; use the offline checks below and leave live verification to the user.

- **Engine unit tests**: `venv/bin/python -m unittest discover -s testing/engine` (run from repo root). ⚠️ Currently all 19 tests error: they patch `engine.STOCKFISH`, a module-level attribute removed in a refactor. Don't treat these failures as caused by your change.
- **Client unit tests**: `venv/bin/python -m unittest discover -s testing/client` — these pass and must stay green. They cover the unreadable-turn fallback and the highlight colour table (both regressions from a live flagged game). To exercise client scan code on real frames, monkeypatch `capture_all_regions` to return the board/clock crops that `save_debug_screenshot` wrote under `logs/sessions/*/errors/` — they are exactly production's inputs.
- **Vision / calibration changes**: run the readback test against saved screenshots:
  `venv/bin/python -m auto_calibration.calibration_readback_test --screenshots auto_calibration/offline_screenshots/desktop --profile desktop`
  It reports FEN extraction, clock OCR, turn/result detection, and per-call timing using production-identical functions.
- **Calibration fitting**: `venv/bin/python -m auto_calibration.offline_fitter --dir <screenshots/> --profile <name> --extract-all --visualise` writes `auto_calibration/calibrations/<name>.json`.
- `main.py --debug` (game-detection dry-run) and `--offline` (replay saved screenshots instead of live capture) exist for when a display is available.

## Key tuning knobs

Behaviour is tuned via `common/constants.py`: `DIFFICULTY`, `QUICKNESS` (move-time pacing), `MOUSE_QUICKNESS` (cursor speed), `RESOLUTION_SCALE` (1.0 for 1080p, 2.0 for 4K). `main.py` flags override some of these per run.

Client-side scan reliability lives in `clients/mp_original.py`: after making a move, `AWAITING_FRESH_SCAN` blocks further moves until a scan is adopted (prevents duplicate/out-of-turn clicks off stale vision), and board scans that fail move-linking or turn detection must survive a confirmation re-capture before being adopted — skipped when our clock is under `RESYNC_CONFIRM_MIN_TIME` seconds (deliberate: humans misread boards under time pressure). When the turn is unreadable, the fallback links the scraped placement from the last tracked fen under *both* turn hypotheses and adopts the fewest-plies candidate — `patch_fens` can "confirm" the wrong turn with a longer piece-shuffle line, which once handed the move to the opponent and flagged a won game (2026-07-12).

## Move-time pacing (`engine.py:_get_time_taken`)

How long the bot "thinks" per move is decided in `Engine._get_time_taken`. A "complicated position" — one worth spending more time on — is keyed off **sharpness** (`self.sharpness`, computed in `_compute_sharpness`): the win-probability spread across the engine's top candidates from a narrow, slightly-deeper scan (multipv 5, depth 12), using the same logistic and definition as the `cheat_detection/` human-likeness analyser. Sharpness ≥ 0.25 is a "critical" position. This replaced the older structural trigger (Lucas `activity`/`eff_mob`); those Lucas metrics are still computed and used elsewhere (mood, resign logic), just no longer for time-scaling. Sharpness maps onto the same `((x+12)/25)**0.4` envelope, so the *magnitude* of the slow-down is unchanged — only the trigger.

An **intuition gate** stops the bot over-thinking (and flagging on time): in a sharp position it applies the slow-down only some of the time and otherwise snaps the move quickly (`base_time *= 0.5`), mirroring that humans trust intuition on most critical moves. Very flat positions (sharpness < 0.10) get a `*0.7` "little at stake" cut. All three log to `engine.log` (sharpness value, multiplier, gate outcome).

**Per-game character** (`_sample_game_character`, resampled whenever `update_info` sees the ply count go backwards — the engine instance persists across games; `new_game()` forces it): draws that vary *between* games what per-move noise can't. `game_pace_sf` (lognormal, scales every think time; applied **after** the phase/mood envelopes because the opening's `**0.2` compression crushes upstream factors; carries a `GAME_PACE_MEAN` 0.90 bias — the 2500-2800 band moves faster; note a mean-pace trim also compresses between-game movetime variance by the same factor). One latent snappiness normal drives `game_premove_sf` (scales every premove-search probability in `make_move`, incl. takeback-only and ponder-dic searches; plus a rating-based snap boost — vs weaker opponents confidence shows up in the premove channel, not think time, because the engine-compute floor means shorter think times can't create <1s moves), `game_ponder_snap_sf` (with pace, via `Engine.ponder_pace_sf`, scales the `ponder_response_wait` — ponder hits are ~30% of moves sitting at the 1s "instant move" boundary), and `game_scramble_fire_sf` (scales the <10s stale-ponder fire probability) — coupled with matching signs because independent draws half-cancel in the realised per-game instant rate. The premove/ponder draws carry deliberate mean biases (`GAME_PREMOVE_MEAN`/`GAME_PONDER_SNAP_MEAN`): mean-1.0 multipliers are *not* mean-preserving in realised rates (probabilities saturate at the top; the instant cutoff is a threshold). The same latent drives `game_ponder_width` (per-game cap on how many opponent replies `ponder()` prepares, clip 0–5; width 0 = a game that never pre-thinks) — the cap only binds because `PONDER_TIME_PER_POSITION` (`common/search_constants.py`, 0.05) stops the ponder budget pinning realised width at 1, which is why width draws did nothing at the old 0.1s/position. Separately: `game_snap_gate` (the intuition-gate probability, Uniform(0.55, 0.95) — raised after the time-vs-sharpness correlation overshot humans) and `game_scramble_skill` (flag-race composure, below). Constants `GAME_*`/`SCRAMBLE_*` in `common/constants.py`. Related strength knob: `HUMAN_EVAL_NOISE_SCALE` (0.75) trims the eval-noise sd at both selection sites — raises top-N engine-match rates toward the 2500-2800 band with no compute cost (unlike a DIFFICULTY bump); diminishing returns below ~0.75. Rationale: cheat_detection's Brown–Forsythe variance test flagged the bot as far too consistent game-to-game — but calibrate targets against innocent *single accounts*, not ratio 1.0 (see cheat_detection below).

**Flag-race autopilot** (`get_stockfish_move`, constants `FLAG_RACE_*`): below 10s of clock, evals in the scramble move-appeal formula are capped — a flagging human reads "+mate" and "+800" both as "winning" and picks by hand-distance and instinct, occasionally missing mates or throwing the win. Without this the bot's endgame-ACPL tail is unhumanly thin (humans: ~20% of games contain a 300+ acpl endgame, tail to 9000+ from thrown mates). The cap is per-game via `game_scramble_skill` u~U(0,1): cap = 550 + u·550, plus a blind-move chance 0.10·(1−u)² that drops evals entirely for one decision (pure hand-distance + noise — that's what actually throws a won ending). The same skill gates the scramble safety vetos with probability 0.75+0.25·u (`SCRAMBLE_VETO_P_*`): `scramble_fire_veto` (common/utils.py; both clients) suppresses blind stale-ponder fires that would drop a piece on a visibly-attacked square, and `get_premove` requires `check_safe_premove` for non-takeback scramble premoves. Both clients also check the **exact ponder hit first in scrambles** (under 10s, if the opponent played the prepared-for move, fire the prepared reply before considering stale fires) — quality instants: 2500-2800 humans fire ~60% of sub-10s moves instantly and those are their *safest* bucket (`python -m cheat_detection.emt_buckets`), while blind fires were the bot's worst. Tuning history (why these exact shapes: absolute vetos collapse the ACPL body/tail, a full 0–1 skill gate floods hang-fires back, and the *appeal-path cap never was the blunder source* — the excess sat in emt-0 instant fires): `cheat_detection/runs/variance_tuning_v1-v5.md`, and the 2500-2800 loop arc in `cheat_detection/runs/ralph/LOG.md` (both gitignored). ⚠️ Twice-proven poison: raising premove *queue* volume (mean bias > ~1.25, or a midgame full-premove channel — even `check_safe_premove`-vetted) re-breaks the TP blunder rate and match rates; queued premoves fire after opponent deviations and after the clock has dropped, beyond any queue-time vet. Widening `SCRAMBLE_FIRE_SF` trades instant-rate variance ~1:1 against TP blunders — the remaining instant-variance residual needs opponent-time pondering (architectural), not more fire volume. The two "close the game" overrides in the same function are deliberately soft (0.4 mate-dash, 0.15 low-piece), and `_decide_human_filters` routes more deep-scramble moves through this fallible path. Costs real win rate in flag races — that's the point.

**Hesitation before the mistake — and the decisive snap** (`_adjust_time_for_move_loss`): after the move is chosen, its win-probability loss is read from the stored deep sharpness scan (`self.sharpness_scan`, depth 12 multipv 5; the 20ms-capped full-width analysis is only a fallback — its evals are too noisy to gate on, which is why the original lever barely worked). Loss ≥ `MISTAKE_HESITATION_WC_LOSS` (0.05): think time stretched ×Uniform(1.4, 2.6) with probability 0.88. Loss ≤ `MISTAKE_SNAP_WC_LOSS` (0.02): trimmed ×Uniform(0.65, 0.9) with probability 0.7 — humans bang out moves they're sure of, and the trim keeps mean move time level despite the stretches. (0.75/0.6 → 0.88/0.7 for the 2500-2800 band's stronger correlation.) Both skipped under 10s of clock (plain clock veto; the old "hurry-mood" veto made the 15s constant unreachable). Restores the human-positive per-game correlation between move time and move loss (humans ≈ +0.11 in the diff150-300 population; measured 0.02 → 0.05–0.06 across validation sims — the engine's errors come from the human-probability sampling and are otherwise independent of think time). `MISTAKE_*` in `common/constants.py`.

## Architecture pointers

- `engine.py` — move selection core (the monolith everything actually runs: live client and simulator both). `engine_components/` is a parallel refactor of the same logic that is **not imported by anything** and has drifted (no sharpness pacing, no per-game character) — don't edit it expecting behaviour changes.
- `common/search_constants.py` — search-breadth and scan-width tuning constants (candidate-move counts, sharpness/premove scan widths, mood breadth deltas) extracted from `engine.py`; documented per-constant in the file.
- `clients/mp_original.py` — the active Lichess client (mouse automation, game loop, scan-reliability guards); other clients in `clients/` are older variants.
- `common/utils.py` — `scraped_fen_sanity_issues()` / `InvalidPositionError`: reject structurally impossible scraped positions (e.g. king hidden by a capture animation); the engine refuses to analyse them (Stockfish segfaults) and the client discards the scan. Two layers: the default checks **placement only** — the raw scrape FEN always claims white to move, so turn-dependent tests would false-positive (a live game froze when we were black in check and every scan was rejected). `turn_reliable=True` — used only in the engine, where `update_info` asserted turn == side — additionally flags `OPPOSITE_CHECK` (side not to move in check), which segfaults Stockfish deterministically and otherwise defeats the crash-restart handler. Don't merge the layers in either direction.
- `chessimage/image_scrape_utils.py` — screen capture, FEN extraction, clock OCR. Last-move highlight detection matches against BGR colours built from `FALLBACK_COLOUR_SCHEME` in `auto_calibration/config.py`; those must never sit within tolerance of Lichess's *selected-square* tint (a stale selection left by a cancelled premove otherwise reads as a last-move highlight and poisons turn detection — guarded by `testing/client/test_highlight_colours.py`).
- `auto_calibration/` — fits board/UI coordinates from screenshots; profiles stored in `auto_calibration/calibrations/`.
- `common/move_timing.py` — the timing formulas (mouse-leg duration, settle sleeps, drag-vs-click probability, ponder/scramble waits, MOVE_DELAY constants) shared between the live client and the simulator. Tune durations here, not inline in the client, or the two drift apart.
- `simulation/` — offline bot self-play with **simulated** clocks (no display/mouse; wall time ≈ engine compute only). `python -m simulation.run --games N --tc 60+0 --workers 5` writes PGNs with integer-second `[%clk]` that feed straight into cheat_detection (`--player SimBotWhite SimBotBlack`); workers parallelise near-linearly (each owns an Engine pair — keep ~cores/3; GPU doesn't help, the cost is CPU Stockfish). The two bots can differ per persona — `--a-*`/`--b-*` set name/rating/difficulty/quickness/mouse (`BotSpec` in `game_runner.py`; per-engine quickness works because `Engine(quickness=...)` overrides the global per instance) — and `--sides alternate` swaps colours each game with correct per-game PGN headers; `--plain` gives line-based progress for the GUI tab. Per-move clock charge = detection + max(time_take − MOVE_DELAY, compute) + mouse gesture, with the live client's fast paths (premove fire, ponder-dic hits, scrambles) reproduced in `client_model.py`. Machine-specific latency distributions come from `python -m simulation.calibrate {detection,compute}` (JSONs in `simulation/calibration/`, gitignored). Drives monolithic `engine.py`, not the unwired `engine_components/` refactor. See its README for the charge model and Phase 2/3 roadmap (sim-vs-real Welch validation). Live sessions default to `--log-level PERF` (`main.py`) so real games accumulate `[PERF]` realised-move/scan/mouse timings for refining the model.
- `cheat_detection/` — offline human-likeness diagnostic (CLI `analyze.py`, Tkinter `gui.py` with per-feature distribution drill-down and a **Simulation** tab that drives `simulation.run` as a subprocess; see its README). Analysis depth defaults to 10 (benchmarked ~24× faster than the original 18). Bot analysis and the baseline JSON must use the same depth — mismatched depths skew every engine-derived feature; baselines store per-game feature values (`values`) for the GUI histograms and the variance test. Baselines/runs/cache/corpora are gitignored. Flagging: effect-size `|z| >= 2` (default) or Welch t-test at `--alpha` (`--test`), plus an independent Brown–Forsythe variance test (⚑v, amber in GUI) that catches un-human game-to-game *consistency* even when means match. ⚠️ Judge ⚑v flags against innocent *single accounts*, not std ratio 1.0: the baseline pools many players, so real accounts are also under-dispersed (innocent medians ~0.90 acpl, 0.46 endgame-acpl, 0.86 instant-rate; calibration write-up in `runs/single_account_calibration.md`). Use `python -m cheat_detection.subsample_check <report.json>` to check flag persistence at innocent-account sample sizes (n≈50) before tuning anything — at n=600 the test has power to flag ratios innocents share. ACPL-family variance ratios also swing heavily between identical-code 300-game sims; never tune them on one run — identical-code replicates showed a noise floor of ±0.1 mean|player_z| and ±1-2 flags per 600-unit run. **Mean-level judging**: don't optimise Welch flags against the pooled baseline — that tests "is the bot the *average* player", which no real account satisfies (innocents with 600 games fail several at p≈1e-10). Instead build a between-player dispersion file (`python -m cheat_detection.player_dispersion --pgn <corpus> --rating 2500 2800 --out baselines/dispersion_2500_2800.json`, minutes on a cached corpus) and judge with `python -m cheat_detection.outlier_check <report.json> --dispersion <json>`: flags |player_z| ≥ 2 or outside the empirical 2.5-97.5% band of real player means (band-only where within-player noise swamps the between signal, e.g. endgame/TP acpl), plus BF flags that persist ≥35% at n=50. Target mean|player_z| ≈ 0.8 (= E|N(0,1)|, what real innocents show) — do **not** drive every feature to 0; too-average is its own tell. Evals cache to SQLite (`cache/analysis_d<d>_mpv<m>.sqlite`, WAL, FEN-keyed) — shared across all time controls and meant to accumulate indefinitely; legacy JSON caches auto-migrate. `--workers N` (CLI + GUI) parallelises analysis game-level (~cores/2 at the default 2 engine threads); only uncached work benefits. Corpus policy: 2300+ only, meaning *at least one* player ≥2300 (`fetch_corpus --any-player --band 2300 + N`; repeatable `--band` fills several quotas in one dump pass), with the off-band opponent excluded at analysis time via `--rating 2300 9999`. Main baseline: `baselines/bullet_1plus0_2300_plus.json` (51.3k units, pure 60+0). Pair-level filters `--opponent-rating` / `--rating-diff` (own−opp, mover-centric) slice a corpus into custom comparison populations — cheap once cached; in `run`/GUI they apply to both humans and bot so the populations stay comparable. Fetches should pin one exact clock with `--tc 60+0` — mixing e.g. 30+0 with 60+0 muddies every timing feature.
